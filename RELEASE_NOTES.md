## v0.9.0 — 휴지통 정리 (clean)

- 새 `clean` 명령: 중복 사본·오래된 파일·선택 분류를 **휴지통으로**(`send2trash`) 보냅니다.
  **영구 삭제는 하지 않으며**, 잘못 보내도 Windows 휴지통에서 복구할 수 있습니다.
- 안전장치(정리와 동일 수준):
  - **Dry-run 기본** — `--confirm-code` 없이는 미리보기만(대상 목록 + 토큰 출력).
  - **계획 토큰 확인** — 실행 시 토큰 일치 필요(웹은 체크박스 + 토큰 입력 2단계).
  - **스캔 폴더 내부로 제한** — `ensure_source_within_scan_root`로 외부 파일 차단.
  - **중복은 그룹당 1개 보존** — 전부 삭제 불가(경로 사전순 첫 파일 보존).
  - 휴지통 이동 내역을 `trash_history_*.json` + 로그에 기록.
- CLI: `clean --trash-duplicates|--trash-old|--trash-category CAT [--confirm-code TOKEN]`.
  웹: **🗑️ 휴지통 정리** 탭.
- 의존성에 `send2trash` 추가. `send2trash` 미설치 시 영구 삭제로 폴백하지 않고 명확히 오류.

## v0.2.0 — 실수 방지 · 결과 신뢰도 · 공유 안전성

### P1. 실행 모드 분리 + 계획 바인딩 확인 토큰
- CLI를 `analyze` / `apply` / `undo` 로 분리. `analyze`는 절대 파일을 이동하지 않습니다.
- `apply`는 `analyze`가 출력한 **계획 토큰**(`--confirm-code`)이 있어야만 실행됩니다.
  - 토큰은 이동 계획(`src->dst` 목록)의 SHA-256 지문이라, analyze 이후 폴더가 바뀌면 토큰이 달라져
    **오래된/미검토 계획이 실수로 적용되지 않습니다.** (고정 문자열 토큰보다 강한 보호)
  - 토큰이 없거나 틀리면 `status: blocked` 로 100% 차단.
- 옛 `run`은 `analyze`의 deprecated 별칭으로만 유지, `run --apply`는 차단.
- Streamlit도 동일한 2단계 게이트(체크박스 + 계획 토큰 입력)로 통일.

### P2. 리포트 버전 관리 + 북마크 마스킹/제외
- 보고서 파일명에 실행 시각 포함: `download_organizer_report_YYYYMMDD_HHMMSS.{md,html,xlsx}`,
  `bookmark_report_YYYYMMDD_HHMMSS.{md,html,xlsx}` — **연속 실행 시 덮어쓰기 없음.**
- `--mask-bookmark-query`: 북마크 URL의 쿼리스트링/프래그먼트 제거. 분석 단계에서 적용되어
  md/html/xlsx 전 산출물과 중복 URL 판정까지 일관되게 마스킹됩니다.
- `--exclude-domain DOMAIN`(반복 가능): 특정 도메인 북마크를 통계/보고서에서 제외.
- 북마크 보고서에 `query masking: on/off` 표기.

### P3. 중복 탐지 모드 (fast / strict)
- `--dup-mode strict|fast` (기본 `strict`). 보고서 Overview에 사용 모드 표기.
- 두 모드 모두 **크기로 1차 그룹핑 후 동일 크기 그룹만 해시**(고유 크기 파일은 해시하지 않음).
  - `strict`: 전체 SHA-256 → 정확, 오탐 없음, 결과 일관성 보장.
  - `fast`: 앞 1MB SHA-256 → 같은 크기·대용량 파일이 많을 때 빠르지만 **과다검출 가능**.
- 성능 트레이드오프: `fast`는 파일당 최대 1MB만 읽고, `strict`는 전체를 읽습니다. 따라서
  같은 크기 후보가 많고 파일이 클수록(>수십 MB) `fast`의 I/O 절감이 커집니다.
  - 중복은 **후보 표시만** 하며 자동 삭제하지 않으므로, fast의 과다검출은 저위험(사용자 검토).

#### 실측 (benchmarks/bench_dup_mode.py, 동일 크기·앞 1MB 동일 데이터셋)

| 데이터셋 | fast 시간 | strict 시간 | 비율 | fast 오탐 | strict |
|----------|----------|------------|------|----------|--------|
| 30개 × 20MB (600MB) | 0.033s | 0.651s | **19.6×** | 30개 표시 | 0개(정확) |
| 200개 × 1MB (200MB) | 0.217s | 0.317s | 1.5× | 200개 표시 | 0개(정확) |

- 결론: **큰 파일이 많을수록 fast의 속도 이득이 큼**(최대 ~20×). 1MB 안팎의 작은 파일
  위주인 일반 다운로드 폴더에서는 차이가 작으므로 **기본값 strict 권장**.
- 재현: `python benchmarks/bench_dup_mode.py --count 30 --size-mb 20`
  (임시 폴더에 합성 파일을 만들어 측정하며, 실제 다운로드 폴더는 건드리지 않습니다.)

### 안전 불변식 (유지)
- 파일 삭제 코드 없음(오직 `shutil.move`). 선택한 스캔 폴더 밖 파일은 이동 직전 재검증으로 차단.
- 되돌리기 시 원위치 점유 파일은 덮어쓰지 않고 건너뜀. 이동은 한 건씩 history에 즉시 기록.

### 검증
- `basedpyright`: 0 errors. `pytest`: 22 passed.
- 최소 테스트 세트: analyze 이동 0건 / apply 토큰 누락·불일치 차단 / 타임스탬프 리포트 덮어쓰기 없음 /
  북마크 마스킹 on·off 비교 / fast vs strict 결과 차이.
